# 2. 인증

## 2-1. 지식 기반 인증(Type 1)

- 사용자가 **알고 있는 정보**를 이용하는 인증 방식
- 예: 비밀번호(Password), PIN, 아이핀(i-PIN)

### 1) 비밀번호(Password)

#### (1) 문제점

- 가장 널리 사용되지만 인증 요소 중 보안성이 낮은 편이다.
- 비밀번호를 암호화하지 않고 전송하면 스니핑으로 도청될 수 있다.
- 높은 수준의 보안을 위해 주기적인 보안 인식 교육이 필요하다.

#### (2) 비밀번호 정책

- 책임 추적성(Accountability)을 위해 비밀번호를 공유해서는 안 된다.
- 로그인 실패 횟수에 임계치(Clipping Level)를 설정한다.
- 로그인과 인증 관련 감사 기록을 정확히 유지한다.
- 비밀번호는 저장 시 단방향 암호화(One-way Encryption)를 적용한다.
- 휴면 계정은 비활성화하거나 삭제한다.
- 신규 계정은 최초 로그인 시 초기 비밀번호를 변경하도록 한다.

---

## 2-2. 소유 기반 인증(Type 2)

- 사용자가 **가지고 있는 것**을 이용하는 인증 방식
- 예: 열쇠, 인증서, 토큰, OTP, 스마트카드

### 1) 스마트카드

- 마이크로프로세서와 메모리를 내장한 소형 컴퓨터 형태의 카드
- 단순 메모리 토큰과 달리 자체적인 처리 능력을 가진다.

#### (1) 문제점

- 국제 표준의 통일성이 부족할 수 있다.
- 전력 소비, 처리 시간, 전자기파 등의 정보를 분석하는 부채널 공격에 취약할 수 있다.

#### (2) 구성 요소

- CPU
- ROM
- RAM
- EEPROM
- I/O 시스템(접촉식·비접촉식)

## 2-3. 존재·생체 기반 인증(Type 3)

- 사용자의 신체적 특징을 비교하여 본인 여부를 인증한다.
- 예: 지문, 정맥, 얼굴, 홍채, 망막
- 물리적 접근 통제와 사용자 식별에 활용할 수 있다.

### 1) 생체인증 분류

1. **신체적 특징(Type 3)**
   - 지문, 홍채, 망막, 얼굴, 손 모양, 정맥 등

2. **행동적 특징(Type 4)**
   - 서명, 키스트로크, 음성, 걸음걸이 등

### 2) 생체인증의 특징

1. **보편성(Universality)**
   - 모든 사람이 해당 생체 특징을 지니고 있어야 한다.

2. **유일성(Uniqueness)**
   - 개인별 특징이 명확하게 구분되어야 한다.

3. **지속성(Permanence)**
   - 생체 정보가 시간의 흐름에도 지속적으로 유지되어야 한다.

4. **성능(Performance)**
   - 개인을 빠르고 정확하게 인식할 수 있어야 한다.

5. **수용성(Acceptability)**
   - 사용자의 거부감이 적고 사용하기 편리해야 한다.

6. **저항성(Circumvention Resistance)**
   - 위조하거나 우회하기 어려워야 한다.

### 3) 생체인증 도구의 효과성과 사회적 수용성

- **효과성:** 손바닥 > 손 > 홍채 > 망막 > 지문 > 목소리
- **수용성:** 홍채 > 키 누름 > 서명 > 목소리 > 얼굴 > 지문 > 손바닥 > 손 > 망막

> 위 순서는 교재와 출제 기준에 따라 표현이 다를 수 있으므로 시험 교재 기준으로 암기한다.

### 4) 생체인증 평가 항목

1. **FRR(False Reject Rate, Type I Error)**
   - 잘못된 거부율
   - 정상 사용자를 비인가자로 판단하여 거부하는 비율
   - 편의성 관점의 지표

2. **FAR(False Acceptance Rate, Type II Error)**
   - 잘못된 승인율
   - 비인가자를 정상 사용자로 잘못 승인하는 비율
   - 보안성 관점의 지표

3. **CER(Crossover Error Rate) / EER(Equal Error Rate)**
   - FAR과 FRR이 같아지는 지점의 오류율
   - 생체인증 시스템의 정확도를 비교하는 척도
   - 값이 낮을수록 성능이 우수하다.

## 2-4. 행동 기반 인증(Type 4)

- 사용자의 반복적인 행동 특성을 프로파일링하여 인증한다.
- 예: 서명, 키스트로크, 마우스 움직임, 걸음걸이, 패턴

### 1) 종류

#### (1) 음성 인식(Voice Recognition)

- **장점:** 원격지에서도 사용할 수 있고 비용이 비교적 저렴하다.
- **단점:** 건강 상태, 감정, 주변 소음 등의 변화에 민감하다.

#### (2) 서명(Signature)

- **장점:** 입력 장치의 가격이 저렴하고 사용 편의성이 높다.
- **단점:** 타인에 의한 모방 가능성이 있으며 상태에 따라 정확도가 낮아질 수 있다.

#### (3) 키보드 입력(Keystroke Dynamics)

- 키를 누르는 속도, 간격, 지속 시간 등의 습관을 이용한다.
- **장점:** 별도의 생체인식 장치 없이 구현할 수 있다.
- **단점:** 입력 과정에서 비밀번호가 노출될 위험이 있으며 공격 대상이 될 수 있다.

## 2-5. 커버로스(Kerberos) 인증

- 중앙집중형 사용자 인증 프로토콜
- 대칭키 암호화 기법을 기반으로 하는 티켓 기반 인증 프로토콜
- 비밀번호를 네트워크로 직접 전송하지 않고 티켓과 세션 키를 사용한다.
- 인증(Authentication)을 제공하며, 시스템 구성에 따라 권한 부여와 계정 관리와 연계될 수 있다.

> **AAA:** Authentication(인증), Authorization(인가), Accounting(계정 관리·사용 기록)

### 1) 구성 요소

1. **KDC(Key Distribution Center)**
   - 키 분배 센터
   - AS와 TGS로 구성된다.
   - 사용자와 서비스의 암호화 키를 관리하고 인증 서비스를 제공한다.

2. **AS(Authentication Server)**
   - 최초 사용자 인증을 수행한다.
   - 인증에 성공한 사용자에게 TGT와 TGS 세션 키를 발급한다.

3. **Principal**
   - Kerberos 인증에 참여하는 사용자 또는 서비스 등의 개체

4. **TGS(Ticket Granting Server)**
   - TGT를 확인하고 특정 서비스에 접근하기 위한 서비스 티켓을 발급한다.

5. **Ticket**
   - 사용자의 신원을 증명하기 위해 사용하는 암호화된 인증 토큰

6. **SS(Service Server)**
   - 사용자가 실제로 이용하려는 서비스를 제공하는 서버

### 2) 동작 원리

1. **사용자 → AS: 최초 인증 요청**
   - 사용자 ID를 KDC의 AS에 보내 최초 인증을 요청한다.

2. **AS → 사용자: TGT와 TGS 세션 키 발급**
   - AS가 TGT(Ticket Granting Ticket)와 사용자–TGS 간 세션 키를 전달한다.

3. **사용자 → TGS: 서비스 티켓 요청**
   - 사용자는 TGT와 인증 정보(Authenticator)를 이용해 원하는 서비스의 티켓 발급을 요청한다.

4. **TGS → 사용자: 서비스 티켓과 SS 세션 키 발급**
   - TGS가 서비스 티켓과 사용자–SS 간 세션 키를 전달한다.

5. **사용자 → SS: 서비스 접속 요청**
   - 사용자가 서비스 티켓과 인증 정보(Authenticator)를 SS에 전달한다.

6. **SS: 티켓 검증 및 서비스 제공 여부 결정**
   - SS가 티켓의 사용자 정보와 유효시간 등을 검증한 후 서비스를 제공한다.

### 3) Kerberos 버전별 특징

| 구분 | Kerberos Version 4 | Kerberos Version 5 |
|---|---|---|
| 암호화 알고리즘 | DES만 사용 | 다양한 암호화 알고리즘 사용 가능 |
| 네트워크 주소 | IP 주소 사용 | 다양한 네트워크 주소 지원 |
| 메시지 표현 | 메시지 바이트 순서 고정 | ASN.1 및 BER 인코딩 규칙 사용 |
| 티켓 유효시간 | 최대 1,280분(28 × 5분) | 시작 시간과 종료 시간으로 유효기간 표현 |
| 상호 인증 | 제한적 | 상호 인증 지원 |
| 암호화 모드 | 비표준 PCBC 모드 사용 | 표준 CBC 모드 사용 |
| 세션 키 | 세션 키를 반복적으로 사용할 수 있어 재생 공격 위험 | 서브 세션 키 사용 가능 |
| 영역 간 인증 | 제한적 | 서로 다른 Realm 간 인증 지원 |

### 4) 핵심

```text
AS  → TGT 발급
TGS → 서비스 티켓 발급
SS  → 티켓 검증 후 서비스 제공
```
---
```text
V4: DES / IP / 고정 / PCBC / 기능 제한
V5: 다양한 암호 / 다양한 주소 / ASN.1·BER / CBC / 기능 확장
```
